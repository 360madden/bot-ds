# Architecture Evidence

## Purpose

This document compares public architecture patterns found in RIFT and other PC MMO automation ecosystems. It records evidence and planning implications without selecting an implementation.

Last reviewed: 2026-07-19.

## Input Scripting And Broadcasting

RIFT-specific AutoHotkey and HotkeyNet material adds an input-only layer that should not be conflated with a state-aware bot.

Observed categories:

- Key remapping and accessibility behavior while the user remains in control.
- Repeated activation of a RIFT in-game macro while a key or toggle is active.
- Multibox broadcasting of one user's input to several RIFT clients.
- Pixel or log reactions that introduce limited state awareness.
- Unattended loops, which cross into autonomous automation.

HotkeyNet and basic AutoHotkey scripts generally provide action output but no authoritative player, target, cooldown, aura, or cast model. They are evidence for actuator behavior and operational controls, not a substitute for a state provider or decision engine.

One notable RIFT project combined an official addon with AutoHotkey and communicated through a few rendered pixels. This independently predates and validates the same broad bridge pattern used by ChromaLink, although its purpose was mouse-look and UI assistance rather than autonomous combat.

Sources are cataloged in `rift-input-scripting.md`.

## Pattern 1: Pixel Observation Plus Synthetic Input

Historical RIFT example: the 2011 AutoIt-tagged pixel bot.

Observed properties:

- Reads visible UI state rather than structured game objects.
- Sends normal keyboard or mouse actions.
- Has low integration cost for simple stationary behavior.
- Is sensitive to resolution, UI scale, occlusion, color changes, and layout.
- Usually has incomplete target, cooldown, aura, and cast information.

Current RIFT example: ChromaLink improves this pattern by having an official addon deliberately render machine-readable pixels. This replaces UI recognition with an explicit optical protocol.

Planning implication: optical telemetry is much more reliable when the addon controls the encoding. The provider still needs frame freshness, sequence numbers, schema versions, and resolution checks.

The local addon API corpus documents UI contexts, positioned and sized frames, numeric RGBA background colors, frame-update events, and frame-relative time. These primitives support the feasibility of an addon-rendered optical protocol, but they do not establish transport reliability or current-client compatibility. See `rift-addon-api-corpus.md`.

## Pattern 2: External Memory Reader Plus Key Output

Historical RIFT example: MMO Goblin / MacroGoblin.

Observed properties:

- Reads structured state such as player health from game memory.
- Keeps action output as configured hotkeys.
- Downloads or updates patch-specific memory-location data.
- Uses ordered ability lists, resource thresholds, health thresholds, combat-log reactions, and key mappings.

Current RIFT example: Reader scans memory for a marker deliberately generated by an official Lua addon rather than reconstructing arbitrary internal game objects.

Planning implication: isolate all memory and transport details behind a provider contract. Patch failure must degrade to "no fresh state" and stop output, never to guessed state.

## Pattern 3: Attached Host Plus Profiles And Behavior Trees

Historical RIFT example: RyftoMate.

Observed artifacts:

- A .NET 4 host attached to the RIFT process.
- XML bot profiles separate from class routines.
- Navigation meshes separate from profiles.
- `TreeSharp.dll` required for routine loading.
- A combat-only XML profile that delegated spellcasting to the selected class routine.

Inference: TreeSharp was widely used by BuddyTeam products as a behavior-tree library. Its presence strongly suggests that RyftoMate composed class and bot behavior as trees. The recovered pages do not expose the actual tree implementation.

Planning implication: preserve the separation among controller mode, combat policy, and game-state provider. A behavior tree is optional; a short priority evaluator may be clearer for the initial combat-only scope.

## Pattern 4: Embedded Script Runtime Over A Native Host

Historical RIFT example: RiftMinion's planned Lua customization.

Current cross-game examples:

- MMOMinion states that its bots are written in Lua over MinionLib and exposes behavior-tree and addon APIs.
- MacroQuest exposes native plugins, its own macro language, Lua scripts, typed game objects, commands, and inter-plugin actors.

Sources:

- <https://wiki.mmominion.com/doku.php?id=lua_api>
- <https://docs.macroquest.org/>
- <https://docs.macroquest.org/main/history-of-macroquest/>

Planning implication: a stable host API plus replaceable scripts can scale well, but embedding a general scripting runtime is unnecessary complexity for an initial data-driven rotation. Profiles should remain declarative unless a concrete requirement exceeds that model.

Important distinction: Lua is only a language. These products' embedded Lua runtimes do not prove compatibility with or equivalence to RIFT's official addon Lua sandbox.

## Pattern 5: Protocol Client Or Proxy

Cross-game example: OpenKore for Ragnarok Online.

Observed properties:

- A long-lived core exposes configuration files, controls, plugins, macros, console commands, and AI sequences.
- Combat conditions are highly data-driven, including skill, resource, target, and safety predicates.
- Network protocol support is a fundamental game adapter concern.

Source:

- <https://openkore.com/>
- <https://openkore.com/wiki/Overview>
- <https://github.com/OpenKore/openkore>

Planning implication: OpenKore validates the value of a normalized state model and declarative conditions. Its protocol-client architecture is not a recommendation for RIFT and should not be imported into this repository without explicit user direction and policy review.

## Repeated Design Lessons

### Stable Core, Replaceable Game Data

MacroGoblin separated patch-sensitive game data from its bot logic. RyftoMate separated profiles, meshes, and routines. RiftMinion planned base class scripts over a shared function API. OpenKore separates core logic, server tables, configuration, and plugins.

Implication: patch-sensitive RIFT observation belongs at the provider boundary. Character builds belong in profiles. Neither belongs in the decision engine.

### Profiles Need Conditions, Not Delays

RiftMinion's developers contrasted the planned full bot with a prototype that required manually entered delays. MacroGoblin separately modeled cooldown timing, health thresholds, resource spenders, pulling, and reactive events.

Implication: profiles should state when an ability is valid, not encode a fixed sleep-based rotation. The engine should use observed GCD, cast, resource, target, cooldown, and aura state.

The local addon API corpus documents current cooldown, cast, resource, target, range, usability, and buff state, but no dedicated global-cooldown field. Whether the reported current cooldown reliably exposes the GCD must be tested rather than assumed.

### Progression Is A Capability Problem

Historical forums organized scripts by calling/class, build, level range, and available skills. Fixed rotations failed as characters learned new abilities or changed souls.

Implication: each profile rule needs explicit availability constraints. The runtime must reconcile the profile with discovered abilities and report unavailable rules rather than silently pressing a key.

The local addon API corpus directly documents available-ability enumeration plus ability add/remove events. This supports discovered capability as the runtime authority while profile level and build requirements remain declarative.

### Combat-Only Is A Valid Independent Mode

RyftoMate distributed a tiny combat-assist XML profile whose only job was to keep the selected class routine active while a human handled PvP or questing.

Implication: combat-only does not require a placeholder movement subsystem. The controller can operate only on the current selected target and remain stopped when no valid target exists.

### Failure Must Be Explicit

Historical support archives are dominated by patch breakage, attach failures, missing routines, stale memory data, unsupported clients, and malformed profiles.

Implication: telemetry status, provider version, schema version, character/profile match, action binding validation, and stop reason should be first-class state. Unknown state must block input.

### Logs Are Part Of The Product

RyftoMate support requested attached logs. MacroGoblin optionally read chat and combat logs. Mature ecosystems expose verbose status, debug commands, and event histories.

Implication: record observed frame identifiers, selected rule, rejected rules, safety vetoes, emitted actions, and stop reasons. Avoid sensitive account or chat content by default.

## Candidate Provider Comparison

| Provider style | State richness | Patch sensitivity | UI sensitivity | Policy/account risk | Current evidence |
| --- | --- | --- | --- | --- | --- |
| Raw screen recognition | Low to medium | Low | High | Automation remains prohibited | Historical pixel bot |
| Addon-rendered optical protocol | Medium to high | Low to medium | Medium | Automation remains prohibited | ChromaLink |
| Addon-generated marker read from memory | High for encoded fields | Medium | Low | Adds process-memory access risk | Reader |
| Arbitrary game-memory reconstruction | High | High | Low | Highest reverse-engineering concern | MacroGoblin history |
| Official addon only | High observation, no verified cast command | Low to medium | Low | Observation is not external automation | RIFT API archive |

This table is evidence for planning. It does not select a provider.

## Source Quality Notes

- Product manuals and source code are best for architecture facts.
- Developer forum posts are strong evidence of intent but weaker evidence of shipped behavior.
- User reviews establish visible behavior and dependencies but rarely prove internals.
- DLL names and runtime dependencies support architectural inference but do not prove full implementation details.
- Search-result snippets are discovery aids only.

## Planning Questions Preserved

- Repository decision (2026-07-19): select providers by technical reliability, maintainability, and privacy. Publisher policy and account-risk tolerance are not implementation criteria for this personal tool.
- Should the first implementation consume an existing Reader or ChromaLink endpoint rather than modify those projects?
- What telemetry update rate and maximum age are required for combat?
- Is a declarative ordered-rule profile sufficient, or is a behavior-tree/runtime scripting layer concretely required?
- How are action-bar bindings validated against discovered abilities?
- What is the exact fail-closed behavior for focus loss, stale telemetry, profile mismatch, death, loading screens, and provider errors?
- Which state recordings can be captured for deterministic offline tests?
- Which fields and events in the version-unknown addon API corpus still conform to the current client, including secure-mode behavior and update latency?

No implementation should begin until these questions are resolved in the approved implementation plan.
